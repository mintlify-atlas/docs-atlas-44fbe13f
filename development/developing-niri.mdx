## Running a Local Build

The main way of testing niri during development is running it as a nested window. The second step is usually switching to a different TTY and running niri there.

<Warning>
Once a feature or fix is reasonably complete, you generally want to run a local build as your main compositor for proper testing. Make sure you know how to revert to a working version in case everything breaks.
</Warning>

The easiest way to run a local build as your main compositor is to:

<Steps>
  <Step title="Install niri normally">
    Install niri from a distro package or other method
  </Step>
  
  <Step title="Overwrite the binary">
    Replace the installed binary with your local build:
    ```bash
    sudo cp ./target/release/niri /usr/bin/niri
    ```
  </Step>
</Steps>

<Tip>
If you use an RPM-based distro, you can generate an RPM package for a local build with `cargo generate-rpm`.
</Tip>

## Logging Levels

Niri uses [`tracing`](https://lib.rs/crates/tracing) for logging. Here's how logging levels are used:

<AccordionGroup>
  <Accordion title="error! - Programming errors and bugs">
    Programming errors and bugs that are recoverable. Things you'd normally use `unwrap()` for. However, when a Wayland compositor crashes, it brings down the entire session, so it's better to recover and log an `error!` whenever reasonable.
    
    <Warning>
    If you see an `ERROR` in the niri log, that **always** indicates a bug.
    </Warning>
  </Accordion>
  
  <Accordion title="warn! - Something bad but possible">
    Something bad but still possible happened. Informing the user that they did something wrong, or that their hardware did something weird, falls into this category. For example, config parsing errors should be indicated with a `warn!`.
  </Accordion>
  
  <Accordion title="info! - Important normal operation messages">
    The most important messages related to normal operation. Running niri with `RUST_LOG=niri=info` should not make the user want to disable logging altogether.
  </Accordion>
  
  <Accordion title="debug! - Less important normal operation">
    Less important messages related to normal operation. Running niri with `debug!` messages hidden should not negatively impact the UX.
  </Accordion>
  
  <Accordion title="trace! - Debugging details (compiled out in release)">
    Everything that can be useful for debugging but is otherwise too spammy or performance intensive. `trace!` messages are **compiled out** of release builds.
  </Accordion>
</AccordionGroup>

## Tests

We have some unit tests, most prominently for the layout code and for config parsing.

<Note>
- When adding new operations to the layout, add them to the `Op` enum at the bottom of `src/layout/mod.rs` (this will automatically include it in the randomized tests)
- When adding new config options, include them in the config parsing test
</Note>

### Running Tests

Make sure to run `cargo test --all` to run tests from sub-crates too.

```bash
cargo test --all
```

Some tests are a bit too slow to run normally, like the randomized tests of the layout code, so they are normally skipped. Set the `RUN_SLOW_TESTS` variable to run them:

```bash
env RUN_SLOW_TESTS=1 cargo test --all
```

<Tip>
It usually helps to run the randomized tests for a longer period, so that they can explore more inputs. You can control this with environment variables:

```bash
env RUN_SLOW_TESTS=1 PROPTEST_CASES=200000 PROPTEST_MAX_GLOBAL_REJECTS=200000 RUST_BACKTRACE=1 cargo test --release --all
```
</Tip>

### Visual Tests

The `niri-visual-tests` sub-crate is a GTK application that runs hard-coded test cases so that you can visually check that they look right. It uses mock windows with the real layout and rendering code.

<Info>
Visual tests are especially helpful when working on animations.
</Info>

To run visual tests:

```bash
cargo run -p niri-visual-tests
```

## Profiling

We have integration with the [Tracy](https://github.com/wolfpld/tracy) profiler which you can enable by building niri with a feature flag.

### On-Demand Profiling (Recommended)

Build with on-demand profiling (data collected only when Tracy is connected):

```bash
cargo build --release --features=profile-with-tracy-ondemand
```

Then you can open Tracy (you will need the latest stable release) and attach to a running niri instance to collect profiling data.

<Tip>
You can run a niri build with on-demand profiling as your main compositor if you'd like.
</Tip>

### Always-On Profiling

<Warning>
When compiled with always-on profiling, niri will **always** collect profiling data, so you can't run a build like this as your main compositor.
</Warning>

If you need to profile niri startup or the niri CLI, use always-on profiling:

```bash
cargo build --release --features=profile-with-tracy
```

### Instrumenting Functions

To make a niri function show up in Tracy, instrument it like this:

```rust
pub fn some_function() {
    let _span = tracy_client::span!("some_function");

    // Code of the function.
}
```

### Memory Allocation Profiling

You can also enable Rust memory allocation profiling:

```bash
cargo build --release --features=profile-with-tracy-allocations
```
